# Fork 工作流指南 - RulesAI/openclaw

## 🎯 仓库结构

```
openclaw/openclaw (官方上游)
  ↓ fork
RulesAI/openclaw (你的Fork - 已创建)
  ↓ clone
本地仓库 (/Users/simon/Github/openclaw)
  ├─ origin → openclaw/openclaw (官方，只读)
  └─ fork → RulesAI/openclaw (你的，可写)
```

## 📦 分支策略

| 分支名                  | 作用                   | 同步到                     |
| ----------------------- | ---------------------- | -------------------------- |
| `main`                  | 官方镜像，保持纯净     | origin/main                |
| `my-local-improvements` | 你的开发分支           | fork/my-local-improvements |
| `feature/*`             | 单独特性分支（用于PR） | fork/feature/\*            |

## 🚀 日常工作流

### 1️⃣ 在开发分支上工作（99%的时间）

```bash
# 确保在开发分支
git checkout my-local-improvements

# 做你的修改...
# 编辑文件、测试、调试

# 提交修改
git add <files>
git commit -m "feat: 描述你的改进"

# 推送到你的Fork（自动备份）
git push
# 或明确指定
git push fork my-local-improvements
```

### 2️⃣ 定期同步官方更新（推荐每周1-2次）

```bash
# 方式A: 一键同步脚本（推荐）
git checkout main && \
git pull origin main && \
git checkout my-local-improvements && \
git merge main && \
git push

# 方式B: 分步执行
git checkout main              # 切换到main
git pull origin main           # 拉取官方更新
git checkout my-local-improvements  # 切回开发分支
git merge main                 # 合并官方更新
git push                       # 推送到你的fork
```

### 3️⃣ 如果遇到合并冲突

```bash
# 合并时提示冲突
git merge main
# CONFLICT (content): Merge conflict in src/file.ts

# 1. 查看冲突文件
git status

# 2. 手动编辑冲突文件，移除标记：
#    <<<<<<< HEAD
#    你的代码
#    =======
#    官方代码
#    >>>>>>> main

# 3. 标记为已解决
git add <resolved-files>

# 4. 完成合并
git commit

# 5. 推送到fork
git push
```

## 🌟 高级操作

### 创建特性分支（准备提PR给官方）

```bash
# 从开发分支创建特性分支
git checkout my-local-improvements
git checkout -b feature/proxy-support

# 只保留你想贡献的提交（交互式rebase）
git rebase -i origin/main

# 推送特性分支到fork
git push fork feature/proxy-support

# 在GitHub上创建PR：
# https://github.com/RulesAI/openclaw → Pull Requests → New
# Base: openclaw/openclaw:main
# Compare: RulesAI/openclaw:feature/proxy-support
```

### 查看你的改进相对官方的差异

```bash
# 查看所有差异
git diff origin/main my-local-improvements

# 只看文件列表
git diff origin/main --name-only

# 查看统计
git diff origin/main --stat
```

### 暂存工作（临时切换任务）

```bash
# 保存当前未提交的工作
git stash save "WIP: 正在做的功能"

# 切换到其他分支工作
git checkout main

# 恢复工作
git checkout my-local-improvements
git stash pop
```

### 重置开发分支到官方版本（谨慎使用）

```bash
# 如果想丢弃所有本地改进，重新开始
git checkout my-local-improvements
git reset --hard origin/main
git push fork my-local-improvements --force
```

## 📋 常用命令速查

```bash
# ========== 查看状态 ==========
git status                              # 查看工作区状态
git log --oneline --graph -10          # 查看提交历史
git branch -vv                         # 查看分支和跟踪关系
git remote -v                          # 查看remote配置

# ========== 分支操作 ==========
git checkout <branch>                  # 切换分支
git checkout -b <new-branch>          # 创建并切换分支
git branch -d <branch>                # 删除已合并分支
git branch -D <branch>                # 强制删除分支

# ========== 提交操作 ==========
git add <files>                       # 暂存文件
git commit -m "message"               # 提交
git commit --amend                    # 修改最后一次提交
git reset --soft HEAD~1               # 撤销最后一次提交（保留修改）

# ========== 同步操作 ==========
git fetch origin                      # 获取官方更新（不合并）
git pull origin main                  # 拉取并合并官方main
git push                              # 推送到跟踪的远程分支
git push fork <branch>                # 推送到指定remote和分支

# ========== 查看差异 ==========
git diff                              # 查看未暂存的修改
git diff --staged                     # 查看已暂存的修改
git diff origin/main                  # 对比官方main
```

## 🎯 实际场景示例

### 场景1: 每天的开发流程

```bash
# 早上开始工作
git checkout my-local-improvements
git status

# 开发、测试...
# 修改文件、运行测试

# 下午提交工作
git add .
git commit -m "feat: 添加新的代理功能"
git push  # 自动推送到 fork/my-local-improvements

# 晚上下班前再次推送（保险）
git push
```

### 场景2: 每周同步官方更新

```bash
# 周一早上同步（一键）
git checkout main && \
git pull origin main && \
git checkout my-local-improvements && \
git merge main && \
git push

# 如果有冲突，解决后：
git add <resolved-files>
git commit
git push
```

### 场景3: 准备提交PR给官方

```bash
# 1. 创建特性分支
git checkout my-local-improvements
git checkout -b feature/add-proxy-support

# 2. 整理提交（可选）
git log --oneline -10
git rebase -i origin/main

# 3. 推送到fork
git push fork feature/add-proxy-support

# 4. 在GitHub上创建PR
# 访问 https://github.com/RulesAI/openclaw
# 点击 "Compare & pull request"
# Base: openclaw/openclaw:main
# Compare: RulesAI/openclaw:feature/add-proxy-support
```

### 场景4: 紧急修复官方Bug

```bash
# 1. 从官方main创建修复分支
git checkout main
git pull origin main
git checkout -b fix/urgent-bug

# 2. 修复bug并提交
git add <files>
git commit -m "fix: 紧急修复xxx问题"

# 3. 推送到fork
git push fork fix/urgent-bug

# 4. 创建PR给官方
# 同时合并到你的开发分支：
git checkout my-local-improvements
git merge fix/urgent-bug
git push
```

## ⚠️ 注意事项

1. **永远不要直接在main分支上开发** - main只用来同步官方
2. **经常提交并推送** - 保护你的工作成果
3. **定期同步官方更新** - 避免累积太多冲突
4. **解决冲突要小心** - 理解你的代码和官方代码的意图
5. **提PR前先测试** - 确保代码可以正常运行
6. **遵守官方的贡献指南** - 查看 CONTRIBUTING.md

## 🆘 故障排除

### 推送被拒绝（diverged）

```bash
# 如果远程分支有新提交，先拉取
git pull fork my-local-improvements --rebase
git push
```

### 误删重要代码

```bash
# 查看所有操作记录
git reflog

# 恢复到某个提交
git reset --hard <commit-hash>
git push --force
```

### 想放弃所有本地修改

```bash
# 重置到远程fork的状态
git fetch fork
git reset --hard fork/my-local-improvements

# 或重置到官方main
git reset --hard origin/main
git push fork my-local-improvements --force
```

### main分支误提交了修改

```bash
# 保存修改到开发分支
git checkout my-local-improvements
git cherry-pick <commit-hash>
git push

# 重置main到官方状态
git checkout main
git reset --hard origin/main
git push fork main --force
```

## 🎓 最佳实践

1. **小步提交** - 每个功能点都提交，方便回滚
2. **有意义的提交信息** - 使用 `feat:`, `fix:`, `refactor:` 等前缀
3. **代码审查** - 提交前用 `git diff` 检查
4. **测试后再推送** - 确保代码能运行
5. **保持分支整洁** - 定期删除已合并的特性分支
6. **备份重要修改** - 推送到远程是最好的备份
